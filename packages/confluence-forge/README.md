# WaveDrom — Confluence Forge app (Cloud)

A Confluence Cloud macro that embeds a WaveDrom timing diagram and opens a
visual editor on click — same UX shape as draw.io. (See "Storage" below — we
use Forge app storage, **not** page attachments, so behaviour around page
copy differs from draw.io.)

## Layout

```
packages/confluence-forge/
├─ manifest.yml                        # Forge module declarations
├─ package.json                        # @forge/api + @forge/bridge deps
├─ src/index.js                        # Forge functions
│                                      #   defaultConfig — mint UUID, seed storage
│                                      #   loadDiagram   — read storage by UUID
│                                      #   saveDiagram   — write storage by UUID
├─ resources/
│  ├─ macro/index.html                 # inline view (SVG preview + Edit button)
│  ├─ macro/bundle/view.js             # generated by `npm run build:forge`
│  ├─ editor/index.html                # full editor modal
│  ├─ editor/bundle/{embed.js,embed.css}  # generated
│  └─ icons/wavedrom.svg               # marketplace icon
```

## Storage

Each macro instance owns a UUID (held in `config.uuid`). The actual WaveJSON
text lives in Forge `storage.set('diagram:<uuid>')` (app-scoped key/value).

**Trade-off vs draw.io**: draw.io stores diagrams as page attachments, so a
page copy carries an independent copy. Here both copies share the storage
entry — editing one mutates the other. Acceptable for now; revisit if users
hit it. Switching to attachments needs `read/write:confluence-content` scopes
and a REST round-trip per load/save.

## View → edit flow

1. **macro/index.html** — read-only inline view.
   - Reads `uuid` from `view.getContext().extension.config.uuid`.
   - `invoke('wavedrom-load', { uuid })` → server returns `{ body }`.
   - Renders SVG via `WavedromView.renderDiagram(body)` (no React, ~10 KB).
   - "✎ Edit" button → opens modal with `new Modal({ resource: 'editor-modal', context: { uuid, initial: body } })`.
   - `onClose` callback re-loads + re-renders.
2. **editor/index.html** — full editor modal.
   - Reads `{ uuid, initial }` from `view.getContext().extension.modal.context`.
   - Mounts the shared React editor (`WavedromEditor.mount`).
   - "Save" → `invoke('wavedrom-save', { uuid, body })` then `view.close()`.

## Theming

`view.getContext().themeState.colorMode` is read on load and mapped to a
`data-theme="dark"|"light"` attribute on `<html>`. The editor's CSS uses
tokens (`--ink`, `--bg`, `--accent`, ...) so the theme switch is one CSS
block.

## Prerequisites (one-time per machine)

1. Node 20+
2. Atlassian developer account — sign up at
   https://developer.atlassian.com (free).
3. Confluence Cloud dev site — create one at
   https://www.atlassian.com/try/cloud/signup?bundle=jira-software&edition=free
   (or use any existing site where you have admin rights).
4. Forge CLI:
   ```sh
   npm install -g @forge/cli
   forge login            # uses an API token from id.atlassian.com/manage-profile/security/api-tokens
   ```

## Build + deploy

From the repo root:

```sh
# 1. Build the shared embed + view bundles and copy them into resources/
npm run build:forge

# 2. First-time only — register the app to get a real app ID, then paste
#    the printed `ari:cloud:ecosystem::app/<uuid>` into manifest.yml
#    (replace REPLACE_WITH_REAL_APP_ID).
cd packages/confluence-forge
forge register           # asks for an app name, e.g. "wavedrom-editor"

# 3. Deploy to the development environment
forge deploy             # builds + uploads functions, pushes Custom UI resources

# 4. Install on your dev site (run once per site)
forge install            # pick "Confluence" → choose your site → confirm scopes
```

Subsequent iterations: `npm run build:forge && forge deploy` from the
package directory. The Custom UI bundle is re-uploaded each time.

## Local development tunnel

```sh
forge tunnel             # streams function logs + serves resources locally
```

`manifest.yml` already declares `tunnel.port: 3000` on `macro-view`; run
`npm run dev` from the repo root in another terminal to serve Vite on 3000
so the tunnel can pick up live changes. (Note: the editor modal isn't
tunneled because Forge modal resources must come from the deployed bundle.)

## Promoting to production

```sh
forge deploy --environment production
forge install --environment production --site <your-site>.atlassian.net
```

Listing on the Atlassian Marketplace requires `forge deploy --environment
production` then submitting via
https://marketplace.atlassian.com/manage/vendors — separate publisher
account from the VS Code marketplace.

## Limitations / risks

- **Custom UI iframe CSP**: editor must be fully self-contained in
  `resources/editor/`. No CDN scripts.
- **Forge invocation quotas**: `loadDiagram` runs on every macro render
  (every page view). For high-traffic pages, consider caching the body in
  config alongside the UUID — last-saved JSON inline, fall back to storage
  read only on demand.
- **PDF / Word export**: Confluence rasterises macros server-side. Inline
  SVG via `WavedromView.renderDiagram()` works because the SVG is in the DOM
  before export crawl; verify after first deploy.
- **Page copy**: see "Storage" above — both copies share storage, edits leak.
